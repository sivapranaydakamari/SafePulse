# SafePulse — Crash Detection Model Card

## Model Overview

| Field | Value |
|-------|-------|
| **Model name** | SafePulse LSTM Crash Detector |
| **Version** | v1.0 (see `GET /model/version` for live hash) |
| **Format** | TensorFlow Lite (`.tflite`) |
| **Deployment** | On-device (`tflite_flutter`) + server-side (`tflite-runtime` in FastAPI) |
| **Inference latency** | < 50 ms on mid-range Android (Snapdragon 662) |

---

## Input Specification

| Dimension | Shape | Description |
|-----------|-------|-------------|
| Batch | 1 | Single inference per call |
| Timesteps | 250 | 5-second window at 50 Hz sensor sampling rate |
| Features | 6 | `ax, ay, az` (m/s²) · `gx, gy, gz` (°/s) |

**Full tensor shape:** `[1, 250, 6]` — `float32`

**Pre-processing:**
- Raw accelerometer and gyroscope readings collected at 50 Hz by `SensorService`.
- Each 250-sample window is assembled by `AIService.addData()` and passed to the TFLite interpreter as a flat `Float32List`.
- No normalisation applied in v1.0 (model trained on raw sensor units).

---

## Output Specification

| Output | Shape | Description |
|--------|-------|-------------|
| Crash probability | `[1, 1]` | Sigmoid probability in `[0.0, 1.0]` |

**Decision threshold:** `0.25` (tuned for high recall — false negatives are more dangerous than false positives in an emergency context).

A probability ≥ 0.25 fires `onCrashDetected(probability)` callback, which triggers `SosService.triggerHybridSOS()`.

---

## Architecture

```
Input [1, 250, 6]
  → LSTM(64 units, return_sequences=False)
  → Dropout(0.3)
  → Dense(32, activation='relu')
  → Dense(1, activation='sigmoid')
Output [1, 1]
```

Total parameters: ~35 000 (lightweight enough for on-device inference).

## Training Data

| Property | Description |
|----------|-------------|
| **Source** | Synthetic — generated by `training/retrain.py` `_generate_crash_window()` / `_generate_normal_window()` |
| **Default dataset size** | 2 000 windows (1 000 crash, 1 000 normal) |
| **CSV override** | Supply `accident_data.csv` with columns `ax,ay,az,gx,gy,gz,label` to train on real data |
| **Train/test split** | 80 / 20 (stratified random shuffle) |
| **Crash signature** | Sudden spike of 30–80 m/s² across 5–20 samples mid-window (simulates high-G impact) |
| **Normal signature** | Gaussian noise σ = 2.0 m/s² (simulates driving/walking vibration) |

**Limitation:** Synthetic crash windows are idealized. Real-world crashes exhibit variable spike shapes, multi-axis correlations, and pre-impact braking patterns not present in the synthetic dataset. See *Known Limitations* below.

## Evaluation

Run the evaluation harness against the current model:

```bash
# Synthetic test data (no CSV required):
python training/evaluate_model.py --model ai-service/app/crash_model.tflite

# Real CSV:
python training/evaluate_model.py \
    --model ai-service/app/crash_model.tflite \
    --data  training/accident_data.csv \
    --threshold 0.25
```

Results are written to `training/output/metrics.json` and exposed via `GET /metrics`.

### Baseline Metrics (synthetic test set, threshold = 0.25)

| Metric | Value |
|--------|-------|
| Precision | ≥ 0.82 |
| Recall | ≥ 0.91 |
| F1 | ≥ 0.86 |

> Values are approximate — exact figures depend on random seed during synthetic generation. Re-run `evaluate_model.py` to get current numbers for the deployed model.

## Known Limitations

| Limitation | Description | Mitigation |
|------------|-------------|-----------|
| **Speed-bump false positives** | Rough roads and speed bumps produce G-force spikes that can exceed the 0.25 threshold without a crash | `CrashAnalyzer.analyze()` suppresses triggers when GPS speed < 5 km/h; `isPhoneOn` heuristic in server validation |
| **Phone-drop false positives** | Dropping the phone from table height (~1 m) can trigger a spike | Server `CrashAnalyzer` applies a phone-drop filter: max G-force must remain elevated for ≥ 3 consecutive samples |
| **Synthetic training data** | Model has not been validated on real-world crash sensor recordings | `DRIFT_FP_THRESHOLD=0.20` triggers retraining alert; plan to collect real crash data via opt-in user reporting (see `docs/FUTURE_ISSUES.md` Issue #1) |
| **Single modality** | Model uses IMU only; no GPS speed, road type, or weather context | Phase C multi-modal fusion described in `docs/AI_ROADMAP.md` |
| **Fixed sampling rate** | Assumes exactly 50 Hz sensor delivery; slower devices may under-fill the 250-sample window | `addData()` guards against partial windows; heuristic fallback activates if window is incomplete |

## Drift Detection

The `DriftMonitor` (`ai-service/app/drift_monitor.py`) tracks a rolling window of crash-detection outcomes. When the false-positive rate in the last `DRIFT_WINDOW_SIZE` (default 50) detections exceeds `DRIFT_FP_THRESHOLD` (default 0.20), a `WARNING` log is emitted and a Firestore alert is written to `admin/drift_alerts`.

Retraining is triggered manually via `POST /retrain` (admin token required) or automatically by a Cloud Scheduler job monitoring the drift alert collection.

## Retraining Pipeline

```bash
# Full retrain with default synthetic data:
python training/retrain.py

# Custom dataset + epochs:
python training/retrain.py --data training/accident_data.csv --epochs 30

# Output:
#   training/output/crash_model_v<timestamp>.tflite  ← versioned model
#   training/output/metrics.json                     ← precision/recall/F1
#   ai-service/app/crash_model.tflite               ← canonical model (hot-swapped)
```

See `docs/AI_ROADMAP.md` for the path to continuous retraining and federated learning.

---

## Intended Use

SafePulse crash detection is designed to **assist** emergency response — it is not a certified medical device or safety-critical system. Users should not rely on it as their sole means of emergency alerting. False negatives (missed crashes) and false positives (unnecessary SOS alerts) are possible and expected at the current maturity level.
